What are Access Control Lists (ACLs)?

Definition and Purpose

An Access Control List (ACL) is a set of rules configured on a router or switch that controls network traffic by permitting or denying packets based on specific criteria.

Core Functions

  • Packet Filtering: Inspect packets based on criteria
  • Security Enforcement: Block unauthorized access
  • Traffic Control: Manage network flow
  • Bandwidth Management: Prioritize traffic types

Filter Criteria

  • Source IP address
  • Destination IP address
  • Protocol (TCP, UDP, ICMP)
  • Port numbers
  • ICMP message types

ACL as a Traffic Cop

Think of ACLs as a security guard checking IDs at a building entrance. The guard has a list of rules: who can enter, who cannot, and under what conditions.

Why Use Access Control Lists?

Security Benefits

1. Block Unauthorized Access

  • Prevent external hosts from accessing internal networks
  • Restrict access to sensitive servers (HR, Finance, Database)
  • Block malicious traffic at network perimeter
  • Implement network segmentation policies

2. Traffic Control and Management

  • Permit or deny specific traffic types (HTTP, FTP, SSH)
  • Allow internal users to access internet while blocking inbound traffic
  • Control routing update propagation
  • Filter ICMP messages to prevent network reconnaissance

3. Quality of Service (QoS) Support

  • Identify and prioritize business-critical traffic
  • Mark packets for QoS policies
  • Reserve bandwidth for voice and video applications

4. NAT Translation Control

  • Define which traffic should be translated by NAT
  • Specify inside/outside addresses for translation
  • Control access to public IP addresses

Defense in Depth

ACLs are a critical layer in the defense-in-depth security strategy, complementing firewalls, intrusion prevention systems, and endpoint security.

Processing Example

Packet from 192.168.1.10 to 203.0.113.5:80 ! ✓ Matches line 1 → PERMIT (processing stops) ! Packet from 192.168.1.20 to 203.0.113.5:80 ! ✓ Matches line 2 → DENY (processing stops) ! Packet from 192.168.1.30 to 203.0.113.5:443 ! ✓ Matches line 3 → PERMIT (processing stops)

Implicit Deny All

Every ACL has an invisible deny ip any any at the end. If no rule matches, the packet is dropped. Always include an explicit permit statement for traffic you want to allow.

Order Matters!

Placing a general rule before a specific rule makes the specific rule unreachable. Always put specific rules first, general rules last.

Standard ACLs

Overview

Characteristics

  • Filter Criterion: Source IP only
  • Number Range: 1-99 and 1300-1999
  • Syntax: Simple, easy to configure
  • Placement: Close to destination

Use Cases

  • Block entire networks or subnets
  • Restrict management access to routers
  • Control VTY line access (SSH/Telnet)
  • Simple traffic filtering scenarios

Standard ACL Syntax

! Basic Syntax access-list [1-99 | 1300-1999] {permit|deny} [source] [wildcard-mask] ! Examples access-list 10 permit host 192.168.1.10 access-list 10 deny 192.168.1.0 0.0.0.255 access-list 10 permit any ! Special Keywords host 192.168.1.10 → Single host (wildcard 0.0.0.0) any → All addresses (wildcard 255.255.255.255)

Why Place Close to Destination?

Standard ACLs filter based only on source IP. If placed too close to the source, they might block legitimate traffic to other destinations. Place them close to the destination to minimize unintended blocking.

Configuration Example

! Scenario: Block 10.1.1.0/24 from accessing 172.16.1.0/24 ! Configure on router closest to 172.16.1.0/24 Router(config)# access-list 1 deny 10.1.1.0 0.0.0.255 Router(config)# access-list 1 permit any ! Apply to interface inbound (traffic entering from 10.1.1.0) Router(config)# interface GigabitEthernet0/0 Router(config-if)# ip access-group 1 in

Wildcard Masks Explained

What is a Wildcard Mask?

A wildcard mask is the inverse of a subnet mask. It tells the router which bits to check (0) and which bits to ignore (1) when matching IP addresses.

Subnet Mask Logic

  • 1 = Network bit (must match)
  • 0 = Host bit (can vary)
  • Example: 255.255.255.0
  • Binary: 11111111.11111111.11111111.00000000

Wildcard Mask Logic

  • 0 = Must match (check this bit)
  • 1 = Ignore (don't care)
  • Example: 0.0.0.255
  • Binary: 00000000.00000000.00000000.11111111

Conversion Formula

Wildcard Mask = 255.255.255.255 - Subnet Mask Examples: ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Subnet Mask Wildcard Mask CIDR 255.255.255.255 → 0.0.0.0 /32 (Single host) 255.255.255.0 → 0.0.0.255 /24 (256 addresses) 255.255.252.0 → 0.0.3.255 /22 (1024 addresses) 255.255.0.0 → 0.0.255.255 /16 (65536 addresses) 255.0.0.0 → 0.255.255.255 /8 (16.7M addresses)

Common Wildcard Patterns

Wildcard Mask Binary Meaning
0.0.0.0 00000000.00000000.00000000.00000000 Match exact host (all bits must match)
0.0.0.255 00000000.00000000.00000000.11111111 Match /24 network (ignore last octet)
0.0.3.255 00000000.00000000.00000011.11111111 Match /22 network (ignore last 10 bits)
255.255.255.255 11111111.11111111.11111111.11111111 Match any address (ignore all bits)

Memory Tip

0 = Obey (must match) | 1 = Ignore (don't care)

Extended ACLs

Overview and Capabilities

Characteristics

  • Number Range: 100-199, 2000-2699
  • Filter Criteria: Source, destination, protocol, port
  • Granular Control: Layer 3 and Layer 4
  • Placement: Close to source

What Extended ACLs Can Filter

  • Source AND destination IP
  • Protocol (TCP, UDP, ICMP, IP, etc.)
  • TCP/UDP port numbers
  • ICMP message types
  • TCP flags (SYN, ACK, FIN, etc.)

Extended vs. Standard ACLs

Feature Standard ACL Extended ACL
Number Range 1-99, 1300-1999 100-199, 2000-2699
Filter Criteria Source IP only Source, destination, protocol, port
Granularity Coarse (all traffic from source) Fine-grained (specific protocols/ports)
Placement Close to destination Close to source
Use Case Block entire networks Block specific services (HTTP, SSH)

Why Place Extended ACLs Close to Source?

Extended ACLs are more specific, so they can filter unwanted traffic early in the path. This conserves bandwidth and router resources by preventing unwanted packets from traversing the network.

Power of Extended ACLs

Extended ACLs provide surgical precision: permit HTTP (port 80) from a specific host to a specific server, while denying all other traffic from that host.

Extended ACL Syntax

Basic Syntax

access-list [100-199 | 2000-2699] {permit|deny} [protocol] [source source-wildcard] [destination destination-wildcard] [port-specification] ┌────────────────────────────────────────────────────────┐ │ [protocol] : ip, tcp, udp, icmp, eigrp, ospf... │ │ [source] : Source IP and wildcard │ │ [destination] : Destination IP and wildcard │ │ [port-spec] : eq, gt, lt, range, neq │ └────────────────────────────────────────────────────────┘

Port Specification Operators

Operator Meaning Example
eq Equal to (single port) eq 80 (HTTP)
gt Greater than gt 1023 (ports above 1023)
lt Less than lt 1024 (well-known ports)
neq Not equal to neq 22 (not SSH)
range Range of ports range 20 21 (FTP)

Practical Examples

! Permit HTTP from any source to web server 203.0.113.10 access-list 100 permit tcp any host 203.0.113.10 eq 80 ! Deny Telnet from 192.168.1.0/24 to any destination access-list 100 deny tcp 192.168.1.0 0.0.0.255 any eq 23 ! Permit HTTPS from host 10.1.1.50 to any destination access-list 100 permit tcp host 10.1.1.50 any eq 443 ! Deny ICMP ping from any to 172.16.0.0/16 access-list 100 deny icmp any 172.16.0.0 0.0.255.255 echo ! Permit SSH from admin subnet to router management interface access-list 100 permit tcp 10.10.10.0 0.0.0.255 host 192.168.1.1 eq 22 ! Permit DNS (TCP and UDP port 53) access-list 100 permit tcp any any eq 53 access-list 100 permit udp any any eq 53 ! Permit FTP (ports 20-21) access-list 100 permit tcp any any range 20 21 ! Permit all other IP traffic access-list 100 permit ip any any

Don't Forget the Permit Statement!

Extended ACLs also have an implicit deny ip any any at the end. Always add an explicit permit statement for traffic you want to allow, or use specific deny statements followed by permit ip any any.

Named ACLs

Advantages of Named ACLs

Benefits

  • Descriptive Names: Use meaningful names instead of numbers
  • Easier Management: Understand ACL purpose at a glance
  • Entry Modification: Insert, delete specific entries
  • Resequencing: Change entry order without recreating
  • Both Types: Standard or extended

When to Use Named ACLs

  • Complex configurations with many entries
  • Need to modify ACL entries frequently
  • Want clear documentation through naming
  • Large enterprise networks
  • Running out of numbered ACL ranges

Named Standard ACL Syntax

! Named Standard ACL Router(config)# ip access-list standard BLOCK_ADMIN_SUBNET Router(config-std-nacl)# deny 192.168.10.0 0.0.0.255 Router(config-std-nacl)# permit any Router(config-std-nacl)# exit ! Apply to interface Router(config)# interface GigabitEthernet0/1 Router(config-if)# ip access-group BLOCK_ADMIN_SUBNET in

Named Extended ACL Syntax

! Named Extended ACL Router(config)# ip access-list extended WEB_SERVER_FILTER Router(config-ext-nacl)# permit tcp any host 203.0.113.10 eq 80 Router(config-ext-nacl)# permit tcp any host 203.0.113.10 eq 443 Router(config-ext-nacl)# deny ip any host 203.0.113.10 Router(config-ext-nacl)# permit ip any any Router(config-ext-nacl)# exit ! Apply to interface Router(config)# interface GigabitEthernet0/0 Router(config-if)# ip access-group WEB_SERVER_FILTER in

Entry Management

! View ACL with sequence numbers Router# show ip access-lists WEB_SERVER_FILTER Extended IP access list WEB_SERVER_FILTER 10 permit tcp any host 203.0.113.10 eq www 20 permit tcp any host 203.0.113.10 eq 443 30 deny ip any host 203.0.113.10 40 permit ip any any ! Delete specific entry Router(config)# ip access-list extended WEB_SERVER_FILTER Router(config-ext-nacl)# no 30 ! Insert entry at specific sequence number Router(config-ext-nacl)# 25 permit tcp any host 203.0.113.10 eq 8080

Best Practice: Use Named ACLs in Production

Named ACLs make configurations self-documenting. Use names like BRANCH_OFFICE_FILTER, DMZ_ACCESS, MGMT_VLAN_CONTROL for clarity.

Applying ACLs to Interfaces

Interface Direction: Inbound vs Outbound

Inbound (IN)

  • When: Packet arrives at interface
  • Before: Routing decision
  • Use Case: Filter traffic entering router
  • Example: Block traffic from untrusted network

Outbound (OUT)

  • When: Packet exits interface
  • After: Routing decision
  • Use Case: Filter traffic leaving router
  • Example: Control access to protected network

Application Syntax

! Apply ACL to interface Router(config)# interface GigabitEthernet0/0 Router(config-if)# ip access-group {ACL-number | ACL-name} {in|out} ! Examples Router(config-if)# ip access-group 10 in Router(config-if)# ip access-group 100 out Router(config-if)# ip access-group WEB_FILTER in

Configuration Rules

One ACL Per Protocol, Per Direction, Per Interface

  • One inbound IPv4 ACL per interface
  • One outbound IPv4 ACL per interface
  • Separate ACLs for IPv4 and IPv6
  • Cannot apply two ACLs in the same direction on same interface

Complete Configuration Example

! Scenario: Block 192.168.10.0/24 from accessing web server 203.0.113.50 ! Use extended ACL close to source ! Step 1: Create extended ACL Router(config)# access-list 100 deny tcp 192.168.10.0 0.0.0.255 host 203.0.113.50 eq 80 Router(config)# access-list 100 deny tcp 192.168.10.0 0.0.0.255 host 203.0.113.50 eq 443 Router(config)# access-list 100 permit ip any any ! Step 2: Apply to interface inbound (traffic from 192.168.10.0) Router(config)# interface GigabitEthernet0/0 Router(config-if)# description Connection to 192.168.10.0 network Router(config-if)# ip access-group 100 in Router(config-if)# exit ! Step 3: Verify configuration Router# show ip interface GigabitEthernet0/0 Router# show access-lists 100

Replacing an ACL

Applying a new ACL to an interface automatically replaces the old one in that direction. The old ACL is removed from the interface but still exists in the configuration.

ACL Processing Order

Sequential Processing Logic

Packet arrives
↓
Check first ACL entry
↓
Match found?
YES ✓
PERMIT or DENY
(Stop processing)
NO ✗
Check next entry
↓
No match found
Implicit DENY ALL

Order Importance Example

Wrong Order (Broken)

access-list 100 permit ip any any access-list 100 deny tcp any host 203.0.113.10 eq 80 ! Problem: Line 1 permits ALL traffic. ! Line 2 NEVER executes (unreachable). ! Result: HTTP to 203.0.113.10 is PERMITTED.

Correct Order

access-list 100 deny tcp any host 203.0.113.10 eq 80 access-list 100 permit ip any any ! Specific rule first: Deny HTTP to server. ! General rule last: Permit all other traffic. ! Result: HTTP to 203.0.113.10 is DENIED.

Critical Rule: Specific Before General

  • Place specific rules (host, single protocol) at the TOP
  • Place general rules (any, permit all) at the BOTTOM
  • Once a match is found, processing STOPS immediately
  • Entries below a matching rule are NEVER evaluated

Implicit Deny All

! Every ACL ends with an invisible implicit deny access-list 100 deny tcp any host 203.0.113.10 eq 80 ! Implicit: deny ip any any (automatically added) ! Without an explicit permit, ALL non-matching traffic is dropped! ! Best practice: Always end with "permit ip any any" if needed

Why Implicit Deny Exists

Security-first design: If you forget to explicitly allow traffic, the default is to deny it. This prevents accidental exposure of networks.

Troubleshooting ACLs

Verification Commands

! Show all ACLs Router# show access-lists ! Show specific ACL Router# show access-lists 100 Router# show ip access-lists WEB_FILTER ! Show ACL applied to interface Router# show ip interface GigabitEthernet0/0 ! Sample output GigabitEthernet0/0 is up, line protocol is up Internet address is 192.168.1.1/24 Outgoing access list is not set Inbound access list is 100 ! Show ACL statistics (match counts) Router# show access-lists Extended IP access list 100 10 deny tcp any host 203.0.113.10 eq www (5 matches) 20 permit ip any any (1523 matches)

Common ACL Mistakes

Mistake Symptom Solution
Wrong direction (in/out) ACL doesn't filter expected traffic Review packet flow, apply correct direction
Implicit deny blocking traffic All traffic dropped Add explicit permit statement
Wrong entry order Specific rules never matched Place specific rules before general rules
Incorrect wildcard mask Wrong hosts matched Recalculate wildcard (255.255.255.255 - subnet mask)
ACL not applied to interface No filtering occurs Use ip access-group command

Testing Methodology

Step-by-Step ACL Testing

  1. Verify ACL exists: show access-lists
  2. Check ACL content: Ensure entries are correct and ordered properly
  3. Verify interface application: show ip interface [interface]
  4. Check direction: Confirm in/out matches packet flow
  5. Test with traffic: Send packets and check match counters
  6. Use logging: Add log keyword to ACL entries for debugging

Debugging ACL Matches

! Add logging to ACL entry Router(config)# access-list 100 deny tcp any host 203.0.113.10 eq 80 log ! View logs Router# show logging %SEC-6-IPACCESSLOGP: list 100 denied tcp 192.168.1.50(52341) -> 203.0.113.10(80) ! Clear ACL counters for fresh testing Router# clear access-list counters 100

Performance Note

The log keyword generates syslog messages for every matched packet, which can impact router performance. Use sparingly and only for troubleshooting.

ACL Best Practices

Configuration Guidelines

1. Document ACL Purpose

  • Use named ACLs with descriptive names
  • Add remark statements to explain purpose
  • Maintain external documentation with change history
Router(config)# ip access-list extended BLOCK_MALICIOUS_TRAFFIC Router(config-ext-nacl)# remark Block known attack sources from threat feed Router(config-ext-nacl)# deny ip 203.0.113.0 0.0.0.255 any Router(config-ext-nacl)# remark Permit all other traffic Router(config-ext-nacl)# permit ip any any

2. Use Named ACLs When Possible

  • Self-documenting configurations
  • Easier to modify specific entries
  • Support for sequence number reordering
  • Better for team collaboration

3. Apply Closest to Traffic Source (Extended ACLs)

  • Conserves bandwidth by dropping packets early
  • Reduces load on downstream routers
  • Faster convergence during network changes

4. Test Before Production Deployment

  • Configure ACL but don't apply to interface immediately
  • Review with show access-lists
  • Test on lab equipment or non-production interfaces
  • Have rollback plan ready (remove ACL command prepared)

5. Regular Audits and Reviews

  • Review ACLs quarterly for outdated rules
  • Check for unused ACLs (show ip access-lists with zero matches)
  • Verify ACL still matches business requirements
  • Remove obsolete entries to improve performance

6. Use Remarks for Inline Documentation

  • Explain why each rule exists
  • Reference ticket numbers or change requests
  • Note business justification

Security Best Practices

Defense in Depth

  • ACLs are ONE layer of security, not the only layer
  • Combine with firewalls, IPS/IDS, endpoint protection
  • Apply ACLs at network perimeter and internal boundaries
  • Filter both inbound and outbound traffic where appropriate

Principle of Least Privilege

Deny by default, permit only what's necessary. Start with deny ip any any mentality, then explicitly permit required traffic.

Advanced ACL Features

1. Time-Based ACLs

Control when ACL rules are active based on time of day or day of week.

! Define time range Router(config)# time-range BUSINESS_HOURS Router(config-time-range)# periodic weekdays 08:00 to 18:00 ! Apply time range to ACL Router(config)# ip access-list extended SOCIAL_MEDIA_BLOCK Router(config-ext-nacl)# deny tcp any any eq 443 time-range BUSINESS_HOURS Router(config-ext-nacl)# permit ip any any

2. Reflexive ACLs

Automatically create temporary return traffic rules for established connections.

! Outbound ACL - Create reflexive entries Router(config)# ip access-list extended OUTBOUND Router(config-ext-nacl)# permit tcp any any reflect TCP_TRAFFIC Router(config-ext-nacl)# permit udp any any reflect UDP_TRAFFIC ! Inbound ACL - Evaluate reflexive entries Router(config)# ip access-list extended INBOUND Router(config-ext-nacl)# evaluate TCP_TRAFFIC Router(config-ext-nacl)# evaluate UDP_TRAFFIC Router(config-ext-nacl)# deny ip any any

3. Object Groups

Group multiple IP addresses, protocols, or ports for easier management.

! Create network object group Router(config)# object-group network WEB_SERVERS Router(config-network-group)# host 203.0.113.10 Router(config-network-group)# host 203.0.113.11 Router(config-network-group)# host 203.0.113.12 ! Create service object group Router(config)# object-group service WEB_SERVICES Router(config-service-group)# tcp eq 80 Router(config-service-group)# tcp eq 443 Router(config-service-group)# tcp eq 8080 ! Use object groups in ACL Router(config)# ip access-list extended WEB_ACCESS Router(config-ext-nacl)# permit tcp any object-group WEB_SERVERS object-group WEB_SERVICES

4. Log Keyword for Monitoring

Generate syslog messages when ACL entries match packets.

! Log denied traffic for security monitoring Router(config)# access-list 100 deny ip any any log ! Log with interval (reduce log volume) Router(config)# access-list 100 deny tcp any any eq 23 log-input ! View logged events Router# show logging %SEC-6-IPACCESSLOGDP: list 100 denied tcp 192.168.1.50(52341) -> 203.0.113.10(23)

Performance Considerations

  • Logging: CPU-intensive, use sparingly in production
  • Reflexive ACLs: Maintain state, consume memory
  • Object Groups: Easier management but slight performance overhead

When to Use Advanced Features

  • Time-based: Control access during specific hours (school networks, parental controls)
  • Reflexive: Stateful filtering without full firewall
  • Object Groups: Managing large ACLs with many similar entries
  • Logging: Security monitoring, compliance auditing, troubleshooting

Summary and Lab Exercises

Key Concepts Recap

  • ACLs filter traffic based on criteria (source, destination, protocol, port)
  • Standard ACLs: Filter by source IP only, place close to destination (1-99, 1300-1999)
  • Extended ACLs: Filter by source, destination, protocol, port; place close to source (100-199, 2000-2699)
  • Named ACLs: Use descriptive names for easier management
  • Wildcard masks: Inverse of subnet masks (0=match, 1=ignore)
  • Processing: Top-to-bottom, first match wins, implicit deny all at end
  • Application: One ACL per protocol, per direction, per interface
  • Placement matters: Standard near destination, Extended near source
